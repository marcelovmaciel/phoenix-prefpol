You are working in a monorepo-like local setup with two Julia packages:

- `Preferences/` is the general package for manipulating candidates, ballots, rankings, profiles, weighted profiles, etc.
- `PrefPol/` depends on `Preferences/` and contains project-specific logic for political-preference analysis over survey data.

Task:
Audit `PrefPol/src/raw_profiles.jl` and implement a careful refactor that backports genuinely general functionality into `Preferences`, while keeping PrefPol’s current behavior and external surface stable.

This is not a greenfield redesign. Do not change functionality. Do not break the current user-facing calls in PrefPol, especially anything touched by `PrefPol/exploratory/`. The point is to improve package boundaries, not to invent a new API for its own sake.

## High-level goals

1. Identify what in `PrefPol/src/raw_profiles.jl` is generic enough to belong in `Preferences`.
2. Move or reimplement that generic functionality in `Preferences`.
3. Simplify `PrefPol/src/raw_profiles.jl` so it mostly orchestrates PrefPol-specific concerns and delegates generic profile-building work to `Preferences`.
4. Preserve current PrefPol behavior and call surface as much as possible.
5. Keep the code simple, readable, and well commented. Prefer explicitness over clever abstractions.

## Very important conceptual boundary

`Preferences` should contain abstractions that make sense outside this project, such as:

- converting tabular score/rating data into weak rankings or profiles
- generic score normalization hooks or generic row-to-ballot logic
- generic profile aggregation / pattern summarization over `Profile` and `WeightedProfile`
- candidate-name normalization helpers only if they are truly domain-generic and not tied to Brazil/ESEB conventions

`Preferences` should NOT absorb PrefPol-specific orchestration such as:

- election year support constants
- loading TOML configs
- scenario lookup by name
- `project_root`-based paths
- assumptions tied specifically to ESEB file layout or PrefPol config machinery
- selection of candidate columns from PrefPol election configs, unless there is a clearly generic core separated from the PrefPol wrapper

* mar 7


You are working in a Julia workspace where:

- `Preferences/` is a general package for preferences, ballots, profiles, and reusable utilities.
- `PrefPol/` depends on `Preferences/`.
- I receive survey datasets with one column per candidate containing candidate scores.
- I want a general utility that, given:
  1. a DataFrame
  2. a vector of candidate-score columns
  3. optionally a weight column
  4. a keyword saying whether to use weights or not

  returns a table with missing percentages for each candidate column.

This functionality is general and should live in `Preferences`, not `PrefPol`.
But I also want a very simple REPL-driven exploratory script in `PrefPol/exploratory` so I can quickly inspect the structure of datasets.

Your task is to implement this in a way that is idiomatic, small, well-documented, and easy to call interactively.

## Functional requirements

Implement a reusable function in `Preferences` that computes, for each candidate-score column:

- candidate/column name
- number of missing observations
- total observations considered
- missing proportion
- missing percentage

It must support:

1. Unweighted mode:
   - missing proportion = `count(ismissing, col) / nrow(df)`

2. Weighted mode:
   - requires a weight column
   - missing proportion = sum of weights for rows where the candidate column is missing divided by sum of weights over all rows included in the calculation
   - return the weighted percentage accordingly

3. Output:
   - return a `DataFrame`
   - one row per candidate column
   - preserve a human-readable candidate/column identifier
   - sort descending by missing percentage by default, unless the local style in `Preferences` suggests otherwise

4. API:
   - accept columns as `Vector{Symbol}` and also work reasonably if user passes strings by converting or throwing a clear error
   - keyword argument for weighted vs unweighted, something like `weighted::Bool=false`
   - keyword argument for `weight_col`
   - use clear argument validation and error messages

## Important conceptual/documentation requirement

In the docstring and comments, explicitly warn that:

- unweighted missingness describes sample completeness
- weighted missingness estimates missingness in the weighted survey population
- survey weights do NOT in general solve item nonresponse bias
- weighted missingness can still be useful descriptively, but should not be overinterpreted

Keep this warning short but explicit. This requirement matters.

## Design requirements

Be smart about where this should live in the project structure.

1. First inspect `Preferences` structure and place the new code where it naturally belongs.
   - If there is already a module/file for survey helpers, tabulation, diagnostics, or dataframe utilities, use it.
   - Otherwise create a small, well-named file rather than stuffing this into an unrelated one.
   - Export the public function from the main module if that matches project style.

2. Keep the implementation minimal.
   - No clever abstractions.
   - No unnecessary submodules.
   - No overengineering.
   - Prefer a straightforward loop or small helper.

3. Follow project conventions.
   - Use the package’s existing style for naming, docstrings, imports, and exports.
   - Add tests if the package already has a test structure.
   - If there are tests, include both weighted and unweighted cases.

## Suggested API shape

Use something close to this unless the project style strongly suggests a better name:

`candidate_missingness_table(df, candidate_cols; weighted=false, weight_col=nothing, sort_desc=true)`

Behavior:

- `candidate_cols` is a vector of column identifiers
- in unweighted mode:
  - `total_observations = nrow(df)`
  - `missing_count = number of missing rows for that candidate`
- in weighted mode:
  - `total_weight = sum(weights over valid rows in df used for denominator)`
  - `missing_weight = sum(weights where candidate value is missing)`
  - include columns named clearly enough that the output makes sense
  - if weighted mode is on, it is acceptable that the “count” fields become weighted totals rather than raw counts, but name them accordingly to avoid ambiguity

A good output schema would be one of these:
- always use generic names like `missing_amount`, `total_amount`, `missing_proportion`, `missing_percent`
- or include raw counts in all cases plus weighted columns only in weighted mode
Choose the cleaner option for the existing package style.

## Edge cases

Handle these explicitly:

- candidate column not found
- weight column missing when `weighted=true`
- nonnumeric or invalid weights
- missing weights
- zero or negative weights

Preferred behavior:
- throw clear errors for structurally invalid cases
- for missing weights, either:
  1. error clearly, or
  2. drop those rows with an explicit documented rule
Choose whichever is more consistent with project style, but be explicit and conservative.

## PrefPol integration requirement

After implementing the reusable function in `Preferences`, add a very simple REPL-driven exploratory script in `PrefPol/exploratory` that demonstrates usage on a dataset.

This script should be easy to run line by line in Emacs/REPL. Keep it in the style of exploratory scripts, not a heavy pipeline.

It should:

- call the new function in unweighted mode
- call it in weighted mode if a weight column exists in the dataset
- print or display the resulting table
- be short and readable
- include a brief header comment explaining purpose and how it fits the project

Do not build a complicated CLI or argument parser.
